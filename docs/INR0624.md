# INR Design 0624

This note records the current compact unified note representation proposal. It is
kept separate from `AnonymousSubmission2027.tex` until the paper text is ready to
adopt the updated design.

## Unified Note Fields

Each note is represented as one unified feature row. Missing fields are filled
with zero and accompanied by field-level presence masks, because zero is a valid
value for timing, pedal, measure offset, and binary notation attributes.

| Part | Symbol | Field | Type | Range / Values | Size | Normalization |
|---|---|---|---:|---:|---:|---|
| meta | $P$ | pitch | binary factors | 12 pitch classes x 8 registers | 12b + 8b | identity |
| meta | $m_{C_s}$ | score control present | binary | 0/1 | 1b | identity |
| meta | $m_{C_p}$ | performance control present | binary | 0/1 | 1b | identity |
| meta | $m_M$ | musical present | binary | 0/1 | 1b | identity |
| score control `C_s` | $I_s$ | score IOI | continuous | 0-5000 ms | 1c | $g(I_s)$ |
| score control `C_s` | $D_s$ | score duration | continuous | 0-5000 ms | 1c | $g(D_s)$ |
| score control `C_s` | $V_s$ | score velocity | continuous | 0-127 | 1c | `x / 127` |
| performance control `C_p` | $I_p$ | performance IOI | continuous | 0-5000 ms | 1c | $g(I_p)$ |
| performance control `C_p` | $D_p$ | performance duration | continuous | 0-5000 ms | 1c | $g(D_p)$ |
| performance control `C_p` | $V_p$ | performance velocity | continuous | 0-127 | 1c | `x / 127` |
| performance control `C_p` | $\mathrm{Ped}_{start}$ | start pedal value | continuous | 0-127 | 1c | `x / 127` |
| performance control `C_p` | $\mathrm{Ped}_{ctrl}$ | pedal control value | continuous | 0-127 | 1c | `x / 127` |
| musical | $\mathrm{md}$ | musical duration | categorical + continuous | top 16 values + 0-6 quarter length | 16b + 1c | exact one-hot + `clip(md / 6, 0, 1)` |
| musical | $\mathrm{ml}$ | effective measure length | categorical + continuous | top 8 values + 0-6 quarter length | 8b + 1c | exact one-hot + `clip(ml_eff / 6, 0, 1)` |
| musical | $m_{\mathrm{ml}}$ | measure length present | binary | 0/1 | 1b | identity |
| musical | $\phi_{\mathrm{mo}}$ | musical onset phase | categorical + continuous | top 16 values + [0, 1) | 16b + 1c | exact one-hot + identity |
| musical | $T_{\mathrm{bpm}}$ | local musical tempo | continuous | 0-300 BPM | 1c | `clip(x, 0, 300) / 300` |
| musical | $\mathrm{Grace}$ | grace | binary | 0/1 | 1b | identity |
| musical | $\mathrm{Hand}$ | hand / staff | binary | 0/1 | 1b | identity |
| musical | $\mathrm{Trill}$ | trill | binary | 0/1 | 1b | identity |
| musical | $\mathrm{Stacc}$ | staccato | binary | 0/1 | 1b | identity |
| musical | $\mathrm{Stem}$ | stem | binary pair | up/down | 2b | identity |

The active EPR musical feature mode is `musical51`. It intentionally follows the
same parameter naming as `perf2midi`: musical onset (`mo`), musical duration
(`md`), and measure length (`ml`). The 51 dimensions are:

```text
md category             16  # top exact md values, no other category
md scalar                1
ml_eff category           8  # propagated effective measure length
ml_eff scalar             1
ml_present                1  # raw ml is present; equivalent to measure boundary
mo_phase category        16  # top exact mo / ml_eff phase values
mo_phase scalar           1
tempo_bpm_norm            1
grace/hand/trill/stacc    4
stem_up/stem_down         2
---------------------------
total                    51
```

Rare values do not use an `other` category. Instead, the corresponding category
block is all zero while the scalar channel preserves the value. The global
musical presence mask $m_M$ still indicates whether the note has musical
features at all; there are no per-parameter missing masks.

## Timing Targets

For EPR, both visible timing controls and timing targets use a fixed log-scale
normalizer:

$$
f_s(t)=\log(1+t/s),
\qquad
g_s(t)=\frac{f_s(\mathrm{clip}(t,0,5000))}{f_s(5000)}.
$$

As of 2026-06-29, the active EPR branch is unified on $s=50$ ms. Shared timing
controls are encoded as $g_{50}(I)$ and $g_{50}(D)$. EPR timing targets are
score-relative differences in the same normalized log domain, with a tighter
deviation range than the earlier $[-1,1]$ target:

```text
ioi_dev_g = g50(perf_ioi_ms) - g50(score_ioi_ms)
ioi_dev_c = clip(ioi_dev_g + 0.5, 0, 1)

duration_dev_g = g50(perf_duration_ms) - g50(score_duration_ms)
duration_dev_c = clip(duration_dev_g + 0.5, 0, 1)
```

These dev values are not stored as visible INR input fields. They are derived
training targets for the EPR timing heads. The visible note schema instead keeps
separate score control and performance control slots, $C_s$ and $C_p$, so the
model does not receive both a performance control and the same information again
as a score-relative dev input.

This keeps IOI and duration unified while reducing the dominance of very long
duration tails. The training loss for timing dev is therefore a loss on this
clipped log-domain target, not a raw millisecond loss. Inference reconstructs
performance timing by inverting the same map:

$$
\Delta \hat g = y - 0.5,
\qquad
g_p = \mathrm{clip}(g_{50}(t_{\mathrm{score}}) + \Delta \hat g, 0, 1),
\qquad
t_{\mathrm{perf}} = 50\left(\exp(g_p f_{50}(5000))-1\right).
$$

### Timing Distribution Heads

The simpler ALN timing head models each timing dev as a single asymmetric
logistic-normal distribution in the normalized log domain:

```text
ALN timing scalar: mu, log_sigma_left, log_sigma_right
```

This keeps the split-normal asymmetry needed for duration / IOI tails while
removing the three-component mixture used by MLN3. The active INR EPR head uses
one timing head for IOI, one timing head for duration, one velocity head, and
four binary pedal logits. There is no split-zero-IOI head and no two-channel
pedal representation.

```text
ioi ALN             4
duration ALN        4
velocity LN         3
pedal binary4       4
pedal_start LN      3
pedal_ctrl LN       3
---------------------
total              21
```

## Timing Relations

For one aligned score-performance note pair, use uppercase feature symbols and
domain subscripts as defined in the field table.

The musical variables reconstruct an absolute score-grid onset from musical
onset (`mo`) and effective measure length (`ml_eff`):

$$
Q = M + \mathrm{mo},
\qquad
I_m = \max(Q - Q_{\mathrm{prev}}, 0),
\qquad
\phi_{\mathrm{mo}} = \left(\frac{\mathrm{mo}}{\max(\mathrm{ml}_{eff}, \epsilon)}\right)\bmod 1.
$$

Chords and same-onset notes have $I_m=0$. This is a score-side structural
condition, not a performance-side zero constraint. The active musical51 input
does not include $I_m$ directly because score IOI $I_s$ already carries local
IOI information; instead it keeps $\phi_{\mathrm{mo}}$ as metrical-position
context.

Before applying the log timing map, the raw score control timing is
approximately induced by musical timing and local tempo. The intermediate
quantity is milliseconds per quarter:

$$
I_s^{\mathrm{ms}} \approx \tau_{\mathrm{ms/q}} I_m,
\qquad
D_s^{\mathrm{ms}} \approx \tau_{\mathrm{ms/q}} \mathrm{md}.
$$

The approximation is exact only when the refined score MIDI follows a locally
constant tempo and the XML-derived score grid aligns perfectly to the refined
score MIDI. In practice, $\tau_{\mathrm{ms/q}}$ can be estimated from either raw
millisecond relation when the denominator is nonzero:

$$
\tau_I = \frac{I_s^{\mathrm{ms}}}{I_m}
\quad\text{if } I_m > 0,
\qquad
\tau_D = \frac{D_s^{\mathrm{ms}}}{\mathrm{md}}
\quad\text{if } \mathrm{md} > 0.
$$

The INR musical tempo feature is stored as BPM rather than raw
milliseconds-per-quarter:

$$
T_{\mathrm{bpm}} =
\frac{60000}{\max(\tau_{\mathrm{ms/q}}, \epsilon)},
\qquad
T_c = \frac{\mathrm{clip}(T_{\mathrm{bpm}}, 0, 300)}{300}.
$$

The 300 BPM cap keeps the normal musical range well resolved and clips the small
right tail caused by very short local segments or score/XML alignment noise. In
a 300-work PianoCoRe sample, about 1.3% of note-level local tempo estimates were
above 300 BPM, while the median was about 116 BPM and the 95th percentile was
about 228 BPM.

This tempo feature is intentionally not used as a multiplicative relation in
the log domain: generally $g(\tau_{\mathrm{ms/q}} I_m) \ne g(\tau_{\mathrm{ms/q}})
I_m$. The model receives BPM only as an auxiliary musical context feature, while
timing controls and EPR timing targets use $g(I)$, $g(D)$, $\Delta I_g$, and
$\Delta D_g$.

The performance timing is represented relative to score control timing:

$$
g(I_p) = \mathrm{clip}(g(I_s) + \Delta I_g, 0, 1),
\qquad
g(D_p) = \mathrm{clip}(g(D_s) + \Delta D_g, 0, 1).
$$

This keeps IOI deviation additive, which handles score chords where $I_s = 0$,
and uses the same centered log-domain additive target for duration.

## Timing Metrics

Saved-MIDI evaluation reports the original millisecond Wasserstein distances
for compatibility and additionally reports fixed log-scale Wasserstein
distances for timing:

```text
ioi_wass, duration_wass          # milliseconds
ioi_log50_wass, duration_log50_wass  # log1p(t / 50), clipped at 5000 ms
```

All log-Wasserstein comparisons use the same $s=50$ scale. Values from
different $s$ settings are not directly comparable because changing $s$ changes
the timing metric space.

## 2026-06-29 Timing Sampling Update

Directly sampling timing dev in the normalized log domain can still create a
large right tail after inverse transformation to milliseconds. The reason is
that the inverse map is exponential:

$$
t = 50\left(\exp(g f_{50}(5000))-1\right).
$$

So a symmetric or slightly biased residual in log space does not remain
symmetric in millisecond space. Positive residuals expand more aggressively
after the inverse map, and duration is especially sensitive because the score
duration prior already has a heavier right tail. This explains why sampled
duration can move right even when the predicted dev distribution looks only
mildly biased.

The current sampling design therefore separates deterministic prediction from
residual sampling. Let $\Delta_{\mathrm{mean}}$ be the deterministic timing
deviation in log space, and let $\Delta_{\mathrm{sample}}$ be the raw sampled
deviation from the model. Instead of directly using $\Delta_{\mathrm{sample}}$,
sample only its residual around the deterministic output:

$$
\epsilon = \Delta_{\mathrm{sample}} - \Delta_{\mathrm{mean}}.
$$

Then apply a small post-hoc asymmetric temperature calibrator:

$$
\tilde\epsilon =
\begin{cases}
\alpha_+ \epsilon, & \epsilon \ge 0,\\
\alpha_- \epsilon, & \epsilon < 0,
\end{cases}
\qquad
\tilde\Delta =
\mathrm{clip}(\Delta_{\mathrm{mean}} + b + \tilde\epsilon, -0.5, 0.5).
$$

Finally decode $\tilde\Delta$ with the same log-scale inverse:

$$
g_p = \mathrm{clip}(g_{50}(t_{\mathrm{score}}) + \tilde\Delta, 0, 1),
\qquad
t_{\mathrm{perf}} = 50\left(\exp(g_p f_{50}(5000))-1\right).
$$

This is still sampling, because the random residual $\epsilon$ remains random.
But the positive and negative residual temperatures are learned separately, so
the sampler can reduce the positive timing tail without making the whole output
deterministic. The calibrator should be fitted on an ASAP calibration split
from training / validation data, not on the ASAP test set.

The provisional CINE $s=50$ ASAP calibration estimated from pooled PP
distributions is:

```text
IOI:
  b        = -0.065
  alpha_+  =  0.581
  alpha_-  =  1.350

Duration:
  b        = -0.130
  alpha_+  =  1.000
  alpha_-  =  0.550
```

On the pooled estimate this moved PP timing Wasserstein from raw sampling toward
the target regime:

```text
IOI W:       deterministic 23.33, raw sample 73.81, calibrated 11.28
Duration W: deterministic 87.73, raw sample 188.38, calibrated 30.87
```

These numbers should be treated as calibration-start values until fitted on a
proper held-out ASAP calibration split. Velocity and pedal sampling are not
changed by this timing calibrator.

## Encoder / Decoder Split

All tasks share the same INR embedding layer. The split between encoder input,
decoder input, and decoder head is only a matter of field masks and target
subsets.

| Task | Encoder Input | Decoder Input | Decoder Head |
|---|---|---|---|
| EPR | current $P$, score control $C_s$, musical | previous $P$, previous performance control $C_p$ | timing dev, velocity, pedal |
| CSR | current $P$, performance control $C_p$ | previous $P$, previous / masked musical | musical |

For EPR, the output layer is no longer one shared continuous head for all
performance fields. The implementation keeps the tensor layout compatible, but
uses separate heads for IOI dev, duration dev, velocity, and pedal-related
outputs. Timing dev remains a head target, not a decoder input field. This
reduces unwanted competition and avoids repeating the same information as both
previous performance control and previous score-relative dev.

The active training protocol also adapts on ASAP for 4 epochs after the main
training stage.

Performance control $C_p$ includes log-scaled timing, velocity, and binary4
pedal. In the current PianoCoRe processed data, `label_shared_raw` and
`label_pedal4_raw` are generated together and checked to have the same note
length, including non-ASAP transcription datasets. Therefore there is no active
mode where performance timing/velocity exists but pedal is absent. If a future
dataset lacks reliable pedal, that should be handled by a loss/data-valid mask
for pedal supervision, not by adding a separate INR mode bit.

The shared input mask is:

$$
\mathrm{mask} = (m_{C_s}, m_{C_p}, m_M).
$$

| Mode | $m_{C_s}$ | $m_{C_p}$ | $m_M$ |
|---|---:|---:|---:|
| EPR encoder | 1 | 0 | 1 |
| EPR decoder | 0 | 1 | 0 |
| CSR encoder | 0 | 1 | 0 |
| CSR decoder | 0 | 0 | 1 |

## Integrated Note Embedding Variants

There are two main integrated note embedding variants:

### SINE: Sum Integrated Note Embedding

SINE maps each feature group to the model dimension and sums the group
embeddings:

$$
h_0 =
E_P(P)
+ m_{C_s} E_{C_s}(C_s)
+ m_{C_p} E_{C_p}(C_p)
+ m_M E_M(M)
+ E_{\mathrm{mask}}(m_{C_s}, m_{C_p}, m_M).
$$

The group masks gate the value embeddings. For example, when $m_{C_p}=0$, the
performance-control value embedding is zeroed and only the explicit mask
embedding records that performance control is absent.

### CINE: Concat Integrated Note Embedding

CINE concatenates the full normalized note schema, including the shared mask,
and maps it to the model dimension with one MLP:

$$
h_0 =
\mathrm{MLP}\left(
\left[
P,\ C_s,\ C_p,\ M,\ 
    m_{C_s},\ m_{C_p},\ m_M
\right]
\right).
$$

Both SINE and CINE use the same field values, the same normalized timing codes,
and the same mask definition. The active EPR implementation uses only the
single log-scale timing control mode, so each score control row has
$(I_s,D_s,V_s)$ and each performance control row has
$(I_p,D_p,V_p,\mathrm{Ped}_{start},\mathrm{Ped}_{ctrl})$. With `musical51`, the
EPR integrated continuous input dimension is
$3 + 5 + 51 + 3 = 62$. The only difference between SINE and CINE is whether
field groups are integrated by summation after group-level embedding or by a
single concat-MLP.

Notes:

- The decoder input is the shifted previous-step INR token, so current-step
  pitch and target fields are masked to avoid leakage. The first decoder step
  uses the BOS/special note token instead of a previous note.
- Encoder-side source fields do not need to be repeated in decoder input. For
  example, EPR decoder input does not repeat score control $C_s$ because it is
  already available through encoder-decoder attention.
- For EPR, keep performance velocity in the head unless you want a
  timing-only variant.
- If the decoder sees musical in CSR, that means query-side fields such
  as pitch, `mo`, `md`, `ml`, and optional binary score attributes, not
  ground-truth outputs for the current note.

## Task Masks

EPR:

- Encoder input: current pitch, score control $C_s$, and available musical.
- Decoder input: previous pitch and previous performance control $C_p$.
- Target: timing dev, performance velocity, and pedal.
- Current-note ground-truth performance fields must be masked from the decoder
  input to avoid target leakage.

CSR:

- Encoder input: current pitch and performance control $C_p$.
- Decoder input: previous pitch and previous / masked musical.
- Target: musical.
- Musical losses should be masked where XML-derived score features are not
  available.
