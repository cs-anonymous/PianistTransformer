# INR Design 0624

This note records the current compact unified note representation proposal. It is

## Unified Note Fields

Each note is represented as one unified feature row. Missing fields are filled
with zero and accompanied by field-level presence masks, because zero is a valid
value for timing, pedal, measure offset, and binary notation attributes.

| Part | Symbol | Field | Type | Range / Values | Size | Normalization |
|---|---|---|---:|---:|---:|---|
| meta | $P$ | pitch | binary factors | 12 pitch classes x 8 registers | 12b + 8b | identity |
| meta | $m_C$ | control present | binary | 0/1 | 1b | identity |
| meta | $\mathrm{control\_is\_perf}$ | control type | binary | score/perf | 1b | ignored when $m_C = 0$ |
| meta | $m_{\mathrm{dev}}$ | dev present | binary | 0/1 | 1b | identity |
| meta | $m_{\mathrm{pedal}}$ | pedal present | binary | 0/1 | 1b | identity |
| meta | $m_M$ | musical present | binary | 0/1 | 1b | identity |
| shared control `C` | $I_{s/p}$ | IOI | binary + continuous | 0-5000 ms | 1b + 1c | `b = x > 500`, `c = x / 500 if !b else x / 5000` |
| shared control `C` | $D_{s/p}$ | duration | binary + continuous | 0-5000 ms | 1b + 1c | `b = x > 500`, `c = x / 500 if !b else x / 5000` |
| shared control `C` | $V_{s/p}$ | velocity | continuous | 0-127 | 1c | `x / 127` |
| dev | $\Delta I$ | IOI deviation | continuous | -500 to 500 ms | 1c | `(x + 500) / 1000`, clipped |
| dev | $\Delta D$ | duration deviation | continuous | -500 to 500 ms | 1c | `(x + 500) / 1000`, clipped |

| musical | $D_m$ | `md` | continuous / ordinal | 0-6 quarter length | 1c | `x / 6` |
| musical | $L$ | `ml` | continuous / ordinal | 0-6 quarter length | 1c | `x / 6` |

| musical | $\mathrm{First}$ | first | binary | 0/1 | 1b | identity |

## Timing Targets

For EPR, timing targets are score-relative rather than absolute log-scale timing:

```text
ioi_dev = perf_ioi_ms - score_ioi_ms
ioi_dev_c = clip((ioi_dev + 500) / 1000, 0, 1)

duration_dev = perf_duration_ms - score_duration_ms
duration_dev_c = clip((duration_dev + 500) / 1000, 0, 1)
```

The current distribution check suggests that `[-500, 500]` covers nearly all IOI
and duration deviations, especially on ASAP. Using the same centered deviation
code for both timing fields keeps the target representation uniform and matches
the current SINE/CINE experiments.

## Timing Relations

For one aligned score-performance note pair, use uppercase feature symbols and
domain subscripts as defined in the field table.

The musical variables reconstruct an absolute score-grid onset:

$$
Q = M + O,
\qquad
I_m = \max(Q - Q_{\mathrm{prev}}, 0).
$$

Chords and same-onset notes have $I_m=0$. This is a score-side structural
condition, not a performance-side zero constraint.

The score-type control timing is approximately induced by musical timing and
local score tempo:

$$
I_s \approx \tau I_m,
\qquad
D_s \approx \tau D_m.
$$

The approximation is exact only when the refined score MIDI follows a locally
constant tempo and the XML-derived score grid aligns perfectly to the refined
score MIDI. In practice, $\tau$ can be estimated from either relation when the
denominator is nonzero:

$$
\tau_I = \frac{I_s}{I_m}
\quad\text{if } I_m > 0,
\qquad
\tau_D = \frac{D_s}{D_m}
\quad\text{if } D_m > 0.
$$

The performance timing is represented relative to score-type control timing:

$$
I_p = \max(I_s + \Delta I, 0),
\qquad
D_p = \max(D_s + \Delta D, 0).
$$

This keeps IOI deviation additive, which handles score chords where $I_s = 0$,
and uses the same centered additive timing target for duration.

The compact absolute timing code is:

$$
I_N =
\begin{cases}
\frac{I}{500}, & I \le 500,\\
\frac{I}{5000}, & I > 500,
\end{cases}
\qquad
D_N =
\begin{cases}
\frac{D}{500}, & D \le 500,\\
\frac{D}{5000}, & D > 500,
\end{cases}
$$

The EPR timing target code is:

$$
\Delta I_N
= \mathrm{clip}\left(\frac{\Delta I + 500}{1000}, 0, 1\right),
\qquad
\Delta D_N
= \mathrm{clip}\left(\frac{\Delta D + 500}{1000}, 0, 1\right).
$$

## Encoder / Decoder Split

All tasks share the same INR embedding layer. The split between encoder input,
decoder input, and decoder head is only a matter of field masks and target
subsets.

| Task | Encoder Input | Decoder Input | Decoder Head |
|---|---|---|---|
| EPR | $P$, score-type $C$, musical | $P$, previous perf-type $C$, previous dev, previous pedal | dev, velocity, pedal |
| CSR | $P$, perf-type $C$, pedal | $P$, previous / masked musical | musical |

The shared input mask is:

$$
\mathrm{mask} = (m_C, m_{\mathrm{dev}}, m_{\mathrm{pedal}}, m_M, \mathrm{control\_is\_perf}).
$$

`control_is_perf` indicates whether the visible shared control $C=(I,D,V)$ is
score-side or performance-side. It is ignored whenever $m_C=0$.

| Mode | $m_C$ | `control_is_perf` | $m_{\mathrm{dev}}$ | $m_{\mathrm{pedal}}$ | $m_M$ |
|---|---:|---|---:|---:|---:|
| EPR encoder | 1 | score | 0 | 0 | 1 |
| EPR decoder | 1 | perf | 1 | 1 | 0 |
| CSR encoder | 1 | perf | 0 | 1 | 0 |
| CSR decoder | 0 | ignored | 0 | 0 | 1 |

## Integrated Note Embedding Variants

There are two main integrated note embedding variants:

### SINE: Sum Integrated Note Embedding

SINE maps each feature group to the model dimension and sums the group
embeddings:

$$
h_0 =
E_P(P)
+ m_C E_C(C, \mathrm{control\_is\_perf})
+ m_{\mathrm{dev}} E_{\Delta}(\Delta)
+ m_{\mathrm{pedal}} E_{\mathrm{pedal}}(\mathrm{Ped})
+ m_M E_M(M)
+ E_{\mathrm{mask}}(m_C, m_{\mathrm{dev}}, m_{\mathrm{pedal}}, m_M, \mathrm{control\_is\_perf}).
$$

The group masks gate the value embeddings. For example, when $m_C=0$, the
shared-control value embedding is zeroed and $\mathrm{type}_C$ is ignored except
inside the explicit mask embedding.

### CINE: Concat Integrated Note Embedding

CINE concatenates the full normalized note schema, including the shared mask,
and maps it to the model dimension with one MLP:

$$
h_0 =
\mathrm{MLP}\left(
\left[
P,\ C,\ \Delta,\ \mathrm{Ped},\ M,\ 
    m_C,\ m_{\mathrm{dev}},\ m_{\mathrm{pedal}},\ m_M,\ \mathrm{control\_is\_perf}
\right]
\right).
$$

Both SINE and CINE use the same field values, the same normalized timing codes,
and the same mask definition. The only difference is whether field groups are
integrated by summation after group-level embedding or by a single concat-MLP.

Notes:

- The decoder input is the shifted previous-step INR token, so current-step
  ground-truth target fields are masked to avoid leakage.
- Encoder-side source fields do not need to be repeated in decoder input. For
  example, EPR decoder input does not repeat score-type $C$ because it is
  already available through encoder-decoder attention.
- For EPR, keep performance velocity in the head unless you want a
  timing-only variant.
- For CSR, the musical head can be reduced further by keeping only one of
  `MO` or `I_m`; if you want the smallest head, I would keep `MO` and derive
  `I_m` from consecutive onsets plus measure structure.
- If the decoder sees musical in CSR, that means query-side fields such
  as `P`, `O`, `I_m`, `D_m`, `L`, and optional binary score attributes, not
  ground-truth outputs for the current note.

## Task Masks

EPR:

- Encoder input: pitch, score-type shared control $C$, and available musical.
- Decoder input: pitch, previous perf-type shared control $C$, previous
  dev, and previous pedal.
- Target: dev, performance velocity, and pedal.
- Current-note ground-truth performance fields must be masked from the decoder
  input to avoid target leakage.

CSR:

- Encoder input: pitch, perf-type shared control $C$, and pedal.
- Decoder input: pitch and previous / masked musical.
- Target: musical.
- Musical losses should be masked where XML-derived score features are not
  available.
